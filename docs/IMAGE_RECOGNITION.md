# 画像認識機能ガイド

Phase 4 で実装されたスクショ認識機能の使い方とテクニカルガイドです。

## 📸 概要

決済アプリの取引リスト画面のスクリーンショットから、複数の取引を一度に認識して登録できる機能です。

### 主な特徴

- **一括認識**: 1 枚の画像から複数の取引を同時に認識
- **高精度**: Gemini Vision API による正確な文字認識
- **重複検出**: 既存の取引との照合で二重登録を防止
- **テンプレート準拠**: 構造化された項目名を自動生成
- **選択的登録**: 不要な取引をスキップ可能

## 🚀 使い方

### 1. 画像を準備

決済アプリの**取引リスト画面**のスクリーンショットを撮影します。

**対応アプリ:**

- 三井住友 OLIVE
- ソニー銀行
- d 払い
- d カード
- PayPay
- その他の決済アプリ

**撮影のコツ:**

- 取引が複数表示されている画面
- 日付、金額、店舗名が見えるように
- できるだけ鮮明に撮影
- 情報が切れていても大丈夫（「??」で補完されます）

### 2. アップロード

1. ダッシュボードまたはトランザクション一覧で「**画像から入力**」をクリック
2. 画像をドラッグ&ドロップ、またはファイルを選択
3. AI が自動的に認識を開始

### 3. 認識結果を確認

認識された取引のリストが表示されます。各取引には以下の情報が含まれます:

- **日付**: 取引日時
- **金額**: 取引金額
- **項目名**: テンプレートに従った構造化された名前
- **カテゴリー**: 推測されたメイン・サブカテゴリー
- **信頼度**: 認識の確実性（高/中/低）
- **重複アラート**: 既存の取引と重複の可能性がある場合

### 4. 登録する取引を選択

- チェックボックスで登録する取引を選択
- 重複の可能性がある取引はデフォルトで非選択
- 必要に応じて選択/解除を切り替え

### 5. 一括登録

「○ 件を登録」ボタンをクリックして選択した取引を一括登録します。

## 🔍 重複検出

以下の条件で既存の取引との重複を検出します:

### 判定基準

1. **日付**: ±1 日以内（重み: 30%）
2. **金額**: ±5%以内（重み: 40%）
3. **店舗名**: レーベンシュタイン距離で類似度計算（重み: 20%）
4. **決済方法**: 決済サービスとのマッチング（重み: 10%）

### 閾値

- **80%以上**: 重複の可能性あり（アラート表示）
- **80%未満**: 別の取引として判定

### アラート表示

重複の可能性がある取引は:

- 黄色の枠で表示
- 「重複の可能性」バッジ
- デフォルトで非選択（チェックなし）
- 理由を説明（「日付が一致、金額が一致しています」など）

## ⚙️ 技術仕様

### 認識フロー

```
画像アップロード
    ↓
Firebase Storage に保存
    ↓
Gemini Vision API で画像解析
    ↓
複数の取引情報を抽出（JSON配列）
    ↓
各取引について:
  - カテゴリー自動推測
  - テンプレートに従った項目名生成
  - 信頼度スコア計算
    ↓
既存取引との重複チェック
    ↓
認識結果リストを表示
    ↓
ユーザーが選択
    ↓
一括登録
```

### 対応ファイル形式

- JPEG (.jpg, .jpeg)
- PNG (.png)
- WebP (.webp)

### ファイルサイズ制限

- 最大: 10MB

### 認識精度

- **高信頼度** (80%以上): 明確に認識できた取引
- **中信頼度** (50-79%): ある程度認識できた取引
- **低信頼度** (50%未満): 不確実な認識

## 🛠️ トラブルシューティング

### 認識がうまくいかない

**原因と対処法:**

1. **画像が不鮮明**

   - より高解像度のスクリーンショットを撮影
   - 画面を明るくしてから撮影

2. **取引が認識されない**

   - 取引リストが画面に表示されているか確認
   - 日付、金額、店舗名が見えているか確認

3. **取引数が少ない**
   - 画面をスクロールして全ての取引を含める
   - 複数枚に分けてアップロード可能

### エラーメッセージ

**「Firebase Storage: User does not have permission」**

- Firebase Storage が有効化されていない
- セキュリティルールがデプロイされていない
- 解決: `firebase deploy --only storage`

**「Gemini API が利用できません」**

- API キーが設定されていない
- 解決: `.env.local`に`NEXT_PUBLIC_GEMINI_API_KEY`を追加

**「取引を認識できませんでした」**

- 画像が取引リストではない
- 画像が不鮮明
- 解決: 別の画像を試す

## 💡 ベストプラクティス

### 効果的な使い方

1. **定期的に一括登録**

   - 週 1 回、決済アプリの履歴をまとめて登録
   - 入力の手間を大幅削減

2. **重複アラートを活用**

   - 既に登録済みの取引はスキップ
   - 二重登録を防止

3. **項目名の「??」を後で修正**

   - 登録後、トランザクション一覧から編集可能
   - AI が学習して次回から精度向上

4. **信頼度の低い取引は確認**
   - 低信頼度の取引は念のため確認
   - 必要に応じて手動修正

### 推奨ワークフロー

```
週末に決済アプリを開く
    ↓
今週の取引リストをスクショ
    ↓
「画像から入力」でアップロード
    ↓
認識結果を確認（30秒）
    ↓
重複をスキップして一括登録
    ↓
完了！（10件でも1分以内）
```

## 🎯 項目名の例

### 実際の認識例

**入力（画像）:**

```
10/7  セブンイレブン  ¥1,280
10/6  オオゼキ       ¥3,450
10/5  電車           ¥200
```

**出力（認識結果）:**

```
1. ??（セブンイレブン）で買い物
   - カテゴリー: 食費 > コンビニ
   - 信頼度: 95%

2. ??（オオゼキ）で食材購入
   - カテゴリー: 食費 > スーパー
   - 信頼度: 92%

3. 電車（??）：?? → ??
   - カテゴリー: 交通費 > 電車（定期外）
   - 信頼度: 85%
```

### 「??」の意味

- **用途が不明**: `??（店舗名）`
- **店舗の詳細が不明**: `スーパー（??）で食材購入`
- **路線が不明**: `電車（??）：駅名 → 駅名`
- **目的地が不明**: `電車（JR）：渋谷 → ??`

**後で手動で修正可能:**

- `??（オオゼキ）で食材購入` → `夕食の材料（オオゼキ）で食材購入`
- `電車（??）：渋谷 → ??` → `電車（JR山手線）：渋谷 → 新宿`
